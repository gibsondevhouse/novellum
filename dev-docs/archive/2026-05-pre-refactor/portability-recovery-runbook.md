# Portability Recovery Runbook

Operational guide for diagnosing and resolving portability backup/restore failures.

> **Status (plan-017 stage-004 phase-005):** The canonical backup format is
> now `.novellum`, generated by `POST /api/backup/projects/[id]` against
> the SQLite database. Restore goes through `POST /api/restore/preview`
> followed by `POST /api/restore/project`. The legacy Dexie-backed
> `.novellum.zip` flow under `src/modules/export/services/portability/`
> is retained for the JSON export workflow only and is never invoked by
> the new SQLite-backed code paths (enforced by
> `tests/lib/legacy-dexie-boundaries.test.ts`).

## Error Code Reference

| Code | Message | Cause | Recovery |
| --- | --- | --- | --- |
| `MISSING_MANIFEST` | Archive does not contain a manifest file. | The ZIP file is not a Novellum backup or is an unrelated archive. | Verify the file is a `.novellum.zip` exported from Novellum. |
| `MALFORMED_MANIFEST` | Manifest file is not valid JSON. | The `manifest.json` inside the archive is corrupted or hand-edited incorrectly. | Re-export the backup from the original browser. |
| `MISSING_REQUIRED_FIELD` | Manifest is missing one or more required fields. | A required manifest field was removed or the archive was created by a non-standard tool. | Re-export from Novellum. Do not manually edit archive contents. |
| `FUTURE_FORMAT_VERSION` | This backup was created with a newer version of Novellum and cannot be imported. | The archive format version is higher than what this app version supports. | Update Novellum to the latest version, then retry import. |
| `FUTURE_DB_SCHEMA_VERSION` | This backup uses a newer database schema and cannot be imported. | The archive was exported from an app with a newer Dexie schema. | Update Novellum to the latest version, then retry import. |
| `INVALID_TABLE_COUNTS` | Manifest contains invalid table count values. | Table counts are negative, non-integer, or otherwise malformed. | Re-export the backup. |
| `INVALID_CHECKSUMS` | Manifest contains invalid checksum entries. | Checksum map has empty strings or invalid entries. | Re-export the backup. |
| `CHECKSUM_MISMATCH` | Archive payload integrity check failed — file may be corrupted. | A payload file's SHA-256 hash doesn't match the manifest. The file may have been modified or corrupted during transfer. | Re-download or re-export the backup. Avoid editing ZIP contents. |

## Common Scenarios

### Scenario: User can't import backup after app update

**Symptoms:** `FUTURE_FORMAT_VERSION` or `FUTURE_DB_SCHEMA_VERSION` error.

**Diagnosis:**

1. Check the manifest's `formatVersion` and `dbSchemaVersion`.
2. Compare against the current runtime values in `zip-export.ts` (CURRENT_FORMAT_VERSION, CURRENT_DB_SCHEMA_VERSION).

**Resolution:** The user needs to update their app. If both sides are on the latest version, this indicates a version mismatch bug — check the schema migration chain.

### Scenario: Backup file corrupted during transfer

**Symptoms:** `CHECKSUM_MISMATCH` error.

**Diagnosis:**

1. Extract the ZIP manually and check `manifest.json` checksums.
2. Re-compute SHA-256 of each `db/*.json` and `kv/localStorage.json` file.
3. Compare against manifest checksums.

**Resolution:** Re-export from the original browser. If the original browser data is also corrupted, check IndexedDB directly via browser dev tools.

### Scenario: Restore completes but data seems incomplete

**Symptoms:** Some entities are missing after restore.

**Diagnosis:**

1. Check the `RestoreResult.tablesRestored` and `rowsRestored` counts.
2. Compare against the manifest's `tableCounts`.
3. Verify the project ID matches.

**Resolution:** Ensure the backup was exported with the correct project selected. Empty tables are expected for unused features (e.g., arcs, story_frames if not configured).

### Scenario: localStorage keys not restored

**Symptoms:** Planning notes or outliner preferences are missing after restore.

**Diagnosis:**

1. Check if the browser restricts localStorage (private browsing mode).
2. Verify `RestoreResult.kvKeysRestored` count.
3. Check `kv/localStorage.json` contents in the archive.

**Resolution:** localStorage restore is best-effort. Keys matching allowlisted prefixes (`novellum:planning:`, `novellum:outliner:`, `novellum:prefs:`) are included. If the browser blocks localStorage, these keys cannot be restored.

---

## Preventive Measures

1. **Regular backups** — Encourage users to export backups periodically, especially before major app updates.
2. **Version pinning** — The manifest embeds exact app and schema versions for traceability.
3. **Checksum verification** — All payloads are SHA-256 verified before any destructive restore action.
4. **Atomic restore** — All DB writes run in a single Dexie transaction. Partial restores cannot occur.
