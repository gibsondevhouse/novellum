# Phase-007 — Notes & Known Limitations

> Captures decisions and limitations that aren't blockers for closing
> F3 / F6 / A2, but are worth tracking so future maintainers (or
> stage-002 release-engineering) can pick them up.

## 1. Multi-project UI restore is the natural follow-up

**Status.** The new `.novellum.zip` produced by
`/api/backup/all` is a zip-of-zips: a top-level `manifest.json`
plus one `projects/<projectId>/<filename>.novellum` per project.
Each inner archive is the existing single-project format that
the SQLite-side restore endpoint (`POST /api/restore/project`)
already accepts.

**What works today.**

- Per-project SQLite round-trip:
  `POST /api/backup/projects/[id]` → `.novellum` → user reopens →
  `POST /api/restore/project` (multipart) → restored.
- Multi-project backup creation:
  `POST /api/backup/all` → `.novellum.zip` → user downloads.

**What's missing.** A UI flow that:

1. Accepts a `.novellum.zip` upload in `ImportBackupDialog`
   (or a sibling component).
2. Reads the top-level `manifest.json` to enumerate the bundled
   projects and (optionally) lets the user pick which to restore.
3. For each picked project, extracts the inner `.novellum` and
   POSTs it to `/api/restore/project`.
4. Reports per-project success / failure, then a summary.

The legacy `ImportBackupDialog` (which still works for the older
Dexie-format backups some users may have on disk) is left in
place; the new multi-project restore should be a separate
component or a new mode in the existing dialog.

**Why not in this phase.** Scope. The single biggest move this
phase makes is taking backup creation from disabled-with-TODO to
working-and-shipping. Adding a parallel restore UI would double
the surface area; doing the restore as a follow-up keeps the
review small and gets backups into users' hands sooner.

## 2. Privacy page is now plain `<pre>`

**Decision.** The privacy page renders `PRIVACY.md` via Vite
`?raw` and shows it as `<pre class="privacy__doc">` with
`white-space: pre-wrap`. The previous page had four hand-styled
`SurfaceCard` sections.

**Why.** Adding a markdown-renderer dependency to V1 just for
the privacy page is overkill. The `?raw` approach gives us
guaranteed parity with `PRIVACY.md` (which is the actual DoD
requirement); rich rendering can be a follow-up if needed.

**Follow-up.** If product wants the styled-section look back, the
right move is to either (a) restructure `PRIVACY.md` so the
inline render looks better (it actually reads cleanly as plain
text already) or (b) add a tiny markdown→Svelte converter and
re-skin the page. Either way, the source of truth stays
`PRIVACY.md`.

## 3. Backup format intentionally extensible

The top-level `manifest.json` schema (`AllProjectsManifest`) has a
`formatVersion: 1` field and a `projects: [{ id, title, archive }]`
inventory. Future versions can:

- Add fields (timestamp granularity, host metadata) without
  breaking older readers (they ignore unknown keys).
- Bump `formatVersion` if the layout changes incompatibly
  (readers can refuse a higher version they don't understand).

The per-project inner archives have their own `manifest.json` +
`checksums.json`, owned by `src/lib/server/backup/manifest.ts` /
`checksums.ts`, which is independent of this outer format.

## 4. `ImportBackupDialog` still uses the legacy Dexie restore

The "Restore from backup file…" button on `/settings/backup`
still wires to the legacy
`src/modules/export/services/portability/restore-service.ts`,
which writes to Dexie (the old browser store), **not** SQLite.
This is the older browser-side backup format from before the
SQLite migration (plan-008 / plan-017).

For users on a fully-migrated SQLite-only install this dialog is
effectively a no-op (the Dexie tables are empty), but it's left
in place because:

- Some users may still have legacy `.novellum.zip` files on disk.
- The new multi-project SQLite restore (item #1 above) is a
  parallel UI, not a replacement.

A clean "two-tier restore" UX (try SQLite first; fall back to
Dexie; warn about format) is the right end state. Out of scope
for V1.

## 5. No tests for `/api/backup/all` yet

Justification: the route is a thin composition of two
already-tested helpers (`buildProjectBackup` is covered by
`src/lib/server/backup/__tests__/`; JSZip is third-party). The
end-to-end behavior (button → endpoint → file download) is best
validated via manual smoke or a Playwright E2E spec. Listed in
phase-001 triage as `needs-manual` for the manual-smoke
checklist; if a follow-up adds an E2E spec for backup creation,
it would naturally cover this route too.
