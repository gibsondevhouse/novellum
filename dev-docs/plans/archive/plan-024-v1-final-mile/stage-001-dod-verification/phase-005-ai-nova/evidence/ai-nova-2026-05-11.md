## End-to-end error flow

For an upstream OpenRouter `401 invalid_key` (user has a bad key):

1. **Upstream provider returns 401.**
   `src/lib/ai/providers/openrouter-provider.ts` (in `complete()`
   or `stream()` pre-flight): the helper `classifyStatus(401)`
   returns `'invalid_key'`. `complete()` throws
   `new AiProviderError('invalid_key', 401, 'OpenRouter complete failed: ...')`;
   `stream()` yields an error chunk with the same `code`/`status`.

2. **Proxy at `/api/ai` catches the typed error.**
   `src/routes/api/ai/+server.ts` `providerErrorResponse(err)`
   returns
   `json({ error: { code: 'invalid_key', message: ... } }, { status: 401 })`.
   The stream branch peeks the iterator's first chunk so the
   pre-stream error becomes a real HTTP 401 response, not a
   200 with an SSE error event after the response is constructed.

3. **Browser client maps to `AppError`.**
   `src/lib/ai/openrouter.ts` `throwMappedProxyError(response, body, surface)`:
   `response.status === 401 && code === 'invalid_key'` →
   `throw new AppError('AI_INVALID_KEY', new Error(message))`.
   `AppError.userMessage` is the friendly text from `error-map`:
   `"Invalid key — your AI API key was rejected. Open Settings → AI to update or re-enter your OpenRouter key."`.

4. **Nova classifier picks up the typed code.**
   `src/modules/nova/utils/classify-nova-error.ts` —
   the `AppError.message` (which is the `userMessage`) contains
   `"Invalid key"` so the existing regex
   `/401|invalid.?key|missingcredentials/i` returns `'invalid_key'`.
   New unit tests in `tests/nova/classify-nova-error.test.ts`
   pin this behavior.

5. **`NovaErrorBoundary` renders the typed copy.**
   `src/modules/nova/components/NovaErrorBoundary.svelte:25-28` —
   `ERROR_COPY.invalid_key` shows the "API key invalid" notice
   with a Retry button.

The `429 rate_limit` flow is identical with `'rate_limit'` /
`AI_RATE_LIMIT` substituted at every hop.
