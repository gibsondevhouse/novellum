# Phase-005 — Notes & Known Limitations

> Captures decisions and limitations that aren't blockers for closing
> F4 / F5 but are worth tracking so future maintainers (or
> stage-002 release-engineering / stage-006 docs-rebaseline) can pick
> them up.

## 1. Error-map text is coupled to the Nova classifier regex — RESOLVED 2026-05-11

**Original decision.** `AI_INVALID_KEY` and `AI_RATE_LIMIT`
messages deliberately included the literal substrings
`"invalid key"` and `"rate limit"` so that
`classify-nova-error.ts`'s regex picked them up.

**Resolution (2026-05-11).** `classify-nova-error.ts` now inspects
`AppError.code` directly via `classifyAppErrorCode()`. The legacy
regex fallback is kept for non-`AppError` inputs. The error-map
text was reverted to the cleaner forms ("Your AI API key was
rejected." / "Too many AI requests.") and `pnpm test` stays at
28 / 28 across the openrouter + classifier suites. See
`impl.log.md` entry [2026-05-11 02:30] for the diff summary.

## 2. Mid-stream errors stay generic

The proxy peeks the **first** chunk of the provider's stream so
**pre-stream** auth/rate-limit failures (the common case) become
real HTTP error responses with a structured `error.code`. Errors
that arrive **mid-stream** (after some delta chunks have already
flowed) come over the SSE event stream as
`data: {"error":{"message":"..."}}` and surface in the browser
client as a plain `Error`, not an `AppError`.

**Why this is OK for V1.**

- Mid-stream auth failures are vanishingly rare in practice (the
  upstream provider rejects unauthenticated requests at the
  start, not in the middle).
- The text-based classifier still catches these errors when the
  upstream message contains "rate limit" / "401" / "MissingCredentials".

**Why this is a follow-up.** If we want full typed-error parity
for mid-stream failures, the SSE error envelope needs to carry
`code` (e.g. `data: {"error":{"code":"rate_limit","message":"..."}}`)
and the browser client's stream loop needs to map it to
`AppError`. About 10 LOC of change spread across the proxy + the
client.

## 3. `MissingCredentialsError` was kept as a distinct class

The DoD only names `INVALID_KEY` and `RATE_LIMIT`; the existing
`MissingCredentialsError` (used for "no key configured at all" —
distinct from "user has a key but it's wrong") was preserved
rather than collapsed into `AppError('AI_KEY_MISSING')`. The
reason is that there are existing UI consumers that
`instanceof`-check `MissingCredentialsError`
([NovaMessageLog.svelte:23](src/modules/nova/components/NovaMessageLog.svelte:23)
uses a regex that includes `/MissingCredentials/i`); collapsing
the class would either break that or require a paired UI change.
The `error-map` does have an `AI_KEY_MISSING` entry that the UI
can use if it migrates later.

## 4. Ollama provider was not updated

The same typed-error pattern can and should be applied to the
Ollama provider (`src/lib/ai/providers/ollama-provider.ts`) so
users who switch to local Ollama see typed errors too. Not done
in this phase because:

- The Ollama provider has different failure modes (daemon not
  running, model not pulled) that don't map cleanly onto
  `invalid_key` / `rate_limit`.
- The proxy's iterator-peek logic at `/api/ai` is provider-agnostic
  — when Ollama starts throwing `AiProviderError`, the proxy
  picks it up automatically.

A separate phase (or a stage-002 release-engineering follow-up)
should add Ollama-specific typed errors (`'daemon_unreachable'`,
`'model_not_found'`, etc.) and route them through the same
`AppError` pipeline.
