**Decisions logged (per stage spec § Decision Log Required):**

1. **Old `+page.svelte`: deleted.** SvelteKit accepts a `+page.ts` with a `throw redirect(...)` load function as a complete route — no sibling `+page.svelte` is required. Deleting eliminates a no-op file and matches the spec's preferred outcome.
2. **Redirect lives in `+page.ts`.** `throw redirect(307, '/settings/appearance')` from `@sveltejs/kit`. Spec confirmed; chosen over an `onMount` `goto` to avoid a flash of intermediate UI and to redirect both client navigations and direct `/settings` loads.
3. **Visual baseline (`settings-shell.png`).** 1280×800 capture of `/settings/appearance` with the PillNav rendered and the Appearance pill highlighted; placeholder body card visible inside the content frame. Confirmed by inspecting Playwright's auto-written snapshot before re-running for the green pass.
4. **PageHeader summary cards: dropped.** The two `summary-card` tiles (Primary Surface / Related Routes) were artifacts of the flat settings page and contradicted the new categorized IA where the PillNav itself communicates the route family. Header now renders only eyebrow / title / description.
5. **Zero behavior change for AI keys & Data Portability.** `<ApiSettings />` is mounted bare under `/settings/ai/+page.svelte` (no wrapping markup, no prop changes — the component itself was not touched and still exports from `$modules/settings`). The Data Portability section under `/settings/data/+page.svelte` is a verbatim copy of the old `<section class="settings-section" aria-labelledby="data-portability-heading">…</section>` markup including the `/settings/migrate` link, with the same scoped CSS rules carried over. Confirmed via the `settings-data-page.test.ts` link href + heading assertions and the relocated `settings.png` baseline still capturing the ApiSettings surface.

**Boundaries diff:** None. `eslint.config.js` unchanged. The `$modules/settings` import from `src/routes/settings/ai/+page.svelte` does not require a new boundary entry — `pnpm run lint` confirms the existing `routes` element rules cover this consumption pattern (`src/modules/settings/**` is currently outside any explicit `module-*` boundary type, which mirrors the prior relative-path import in the deleted `+page.svelte`). No new entry added.
